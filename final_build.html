Final project: two-part build

Your final project has two parts: (1) a group website redesign that builds on your midterm analysis, and (2) an individual personal website you can use beyond this class. Both must be live and published.

Group Project

Website Redesign (Continuation of Midterm)

In groups of up to 4, redesign the same website you evaluated for your midterm. Your goal is to deliver a clearly improved, professional version.

What to redesign

  • The same business/nonprofit/approved site from your midterm
  • Your redesign must show clear, meaningful improvement

Requirements

  • No commercial templates (inspiration is fine). Bootstrap allowed.
  • Valid HTML + CSS with clean, readable code
  • Responsive across desktop, tablet, and mobile (no horizontal scroll)
  • Organized files/folders (HTML, CSS, images), no broken links
  • Clear, consistent naming (no spaces in file names)
  • Published live via GitHub Pages or CodePen (multi-page)

Show improvement

  • Stronger visual design and layout
  • Better navigation and information structure
  • Clearer, more readable content
  • Improved spacing, hierarchy, and imagery
  • Better mobile experience

What to submit (one group submission)

  • Bullet-point summary of improvements
  • Live site link (GitHub Pages or CodePen)
  • ~5 minute video: problems → changes → improvements

Grading (30 pts)

  • Quality & usability (10)
  • Technical quality (10)
  • Improvement vs. original (10)
Individual Project

Personal Website

Build a personal website you can use for internships, jobs, or showcasing your work. By the end, your site will be live online.

What it can be

  • Personal site or professional portfolio
  • Profile (education/experience)
  • Projects, interests, blog, or a cause you care about

Requirements

  • HTML + CSS (Bootstrap allowed; no commercial templates)
  • At least 5 pages with different layouts
  • Consistent header/footer across pages
  • Responsive (no horizontal scrolling)
  • Clean, properly formatted code
  • Organized folders; no broken links; good file naming
  • Published live via GitHub Pages or CodePen (multi-page)

Content expectations

  • Combine text, images, and links for an engaging experience
  • Avoid empty pages, only bullet lists, or large unstructured text
  • Make it something you would actually show an employer

Grading (30 pts)

  • Quality & usability (10)
  • Technical quality (10)
  • JavaScript features (10)

What to submit

  • Live site link (GitHub Pages or CodePen)
Before you submit
  • Test your live link in an incognito/private window
  • Check multiple screen sizes
  • Fix broken links and layout issues
  • Review your video (group project)

What matters most in modern web development?

Modern web development is more than getting code to work. It means building websites that are readable, responsive, interactive, accessible, organized, and ready to publish.

Foundation

Semantic HTML

Use meaningful elements like header, nav, main, section, and footer so your content has structure and purpose.

Why employers care: Good structure improves accessibility, maintainability, and professionalism.

Design

Responsive Layouts

Your site should work on phones, tablets, and desktops instead of only looking right on your own screen.

Why employers care: Modern teams expect mobile-friendly interfaces by default.

Layout

Flexbox and Grid

Use modern CSS layout tools to create clean, adaptable page structures.

Why employers care: Layout skill is one of the most visible front-end strengths.

Interaction

JavaScript

Use JavaScript to make your site respond to clicks, inputs, forms, menus, and dynamic changes.

Why employers care: Interactivity turns static pages into usable experiences.

Inclusion

Accessibility

Think about readable text, alt text, keyboard use, contrast, and logical structure.

Why employers care: Accessible design is increasingly expected in professional work.

Publishing

Version Control + Deployment

A strong project is organized, backed up, and shareable through a live link.

Why employers care: Git and deployment workflows are part of real web development practice.

CodePen vs. GitHub Pages

Both can work. The best choice depends on whether you want the fastest route or the more portfolio-ready workflow.

Low friction

CodePen

Best for: Fast prototyping, quick demos, smaller front-end builds, and students who want less setup.

  • Very easy to start
  • Live preview updates quickly
  • Great for experimenting with HTML, CSS, and JavaScript
  • Helpful when your project is mainly one page or a feature demo

Tradeoffs

  • Can feel limiting for larger multi-page projects
  • Not the strongest fit for learning a Git-based workflow
  • Some expanded features depend on paid plans
Portfolio-ready

GitHub Pages

Best for: Multi-page sites, portfolios, stronger file organization, and more realistic publishing practice.

  • Great for static final websites
  • Builds familiarity with repositories and file structure
  • Easy to share as a live project URL
  • Closer to how many developers publish and manage front-end work

Tradeoffs

  • Slightly steeper learning curve at the beginning
  • Students must manage folders, assets, and naming carefully
  • Better for complete projects than quick experiments

New to GitHub?

Beginner-friendly GitHub Pages help

If GitHub Pages feels new, start small. You do not need advanced Git commands for this class. Focus on creating a repository, adding your files, and turning on Pages.

  1. Create a new repository with a simple name
  2. Upload your index.html, CSS, JavaScript, and images
  3. Commit your files
  4. Open Settings → Pages
  5. Select the branch/folder to publish
  6. Open your live link and test it

Step-by-step publish paths

Pick the path that matches your recommendation or your comfort level.

Best for multi-page and portfolio-ready sites

GitHub Pages walkthrough

  1. Create a GitHub account if you do not already have one.
  2. Create a new repository with a clear project name.
  3. Add your files, including index.html, CSS, JavaScript, and image folders.
  4. Check your file names carefully. Spacing, capitalization, and folder paths matter.
  5. Commit your changes with a useful message.
  6. Open repository settings and turn on GitHub Pages.
  7. Choose the branch and folder GitHub Pages should publish from.
  8. Wait for the site to build, then test the live URL on desktop and mobile.
Best use case A complete final project with multiple pages, navigation, assets, and stronger portfolio value.
Common mistakes Missing index.html, broken image paths, wrong file capitalization, and forgetting to republish after changes.
Why this matters This route gives you practice with file structure, publishing, and a more realistic development workflow.
Best for fast and simple front-end builds

CodePen walkthrough

  1. Create a CodePen account.
  2. Start a new Pen and separate your HTML, CSS, and JavaScript into the correct panels.
  3. Build and preview as you go using the live update.
  4. Keep your code clean and organized even if the setup feels easier.
  5. Test your design carefully for spacing, responsiveness, and readability.
  6. Save your Pen and make sure the sharing settings match your assignment needs.
  7. Submit the public or shareable link as your final project.
Best use case A one-page site, front-end demo, creative interaction, or smaller project where speed matters most.
Common mistakes Forgetting mobile checks, stuffing too much into one page, and assuming easy setup means less attention to quality.
Why this matters This route helps you focus on front-end thinking without the extra overhead of repository setup.
Worth knowing

Other modern publishing options

You may also hear about tools like Netlify and Vercel. These are modern deployment platforms often used for front-end websites and web apps. They are powerful, but for this class, GitHub Pages and CodePen are probably the best fit because they keep the learning curve more manageable.
